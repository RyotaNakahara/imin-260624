# 予定調整 Web アプリ
## どう作ったか、何に気を付けたか

社内エンジニア向け発表
imin-260624

---

## このプロダクトは何か

- ログイン不要で予定調整ができる Web アプリ
- ホストが候補日を提示 → ゲストが URL から出欠回答
- Doodle / 調整さん に近い UX
- 最大の特徴：**アカウント登録・ログインが一切不要**

---

## 主な機能

- **ホスト**：タイトル・説明・候補日（日付 or 日時）・回答期限を設定して予定を作成
- **ゲスト**：URL を開いて表示名を入れるだけで回答
  - 出席可 / 出席不可 / 未定 ＋ コメント
- **ホスト**：管理用 URL から回答状況を一覧・集計で確認、後から編集も可能

---

## アクセス制御の設計
### 一番気を付けたところ

ログインを無くす代わりに、**3 種類のトークン**で権限を分離

| トークン | 役割 | 長さ |
|---|---|---|
| `eventId` | ゲストの閲覧・回答用 | nanoid 21文字 |
| `hostToken` | ホスト管理用 | nanoid 32文字 |
| `responseToken` | ゲスト自身の再編集用（Cookie保存） | nanoid 32文字 |

---

## アクセス制御で気を付けたこと

- 存在しない `eventId` ／ 間違った `hostToken` は **すべて同じ 404**
  → 「IDはあるが権限がない」という情報を漏らさない
- ホスト管理画面の API レスポンスから、ゲストの `responseToken` は **返さない**
  → レビューで「ホストが他人の回答を書き換えられる」リスクに気付いて対応
- 回答期限切れはサーバー側でも 403 を返す
  → クライアントのチェックだけに頼らない

## 技術スタック

| レイヤ | 選定 | 理由 |
|---|---|---|
| Next.js 16 (App Router) | UIとAPIを1リポジトリで完結 |
| TypeScript + Zod | API/フォーム入力のスキーマを共通化 |
| Prisma + SQLite | MVPとして軽量。マイグレーションで管理 |
| date-fns / date-fns-tz | JST固定の日時変換を一元化 |
| Vitest | ユニット＋API統合テストを同じツールで |

## 実装の進め方

1. 設計ドキュメント（`docs/design.md`）を先に作成
2. 基盤セットアップ（Next.js + Prisma + SQLite）
3. 予定作成API・ゲスト回答・作成フロー画面
4. ホスト管理画面・予定編集API
5. 再回答モード（Cookieトークンで前回回答を復元）
6. テスト追加（69→71件）
7. カレンダー選択UI・集計サマリー・コメント欄／「未定」を追加

> コアが動く状態を早く作り、UI・細部は後から拡張

---

## テストで気を付けたこと

- ロジック（日時変換・トークン生成・重複チェック等）は `lib/` に切り出し
  → UI/APIに依存しないユニットテスト
- API は実際に Prisma 経由で SQLite に対し統合テスト
  → モックでお茶を濁さない
- テスト用DBは本番用と分離（`test.db`）

## ハマったところ（1）

### Prisma Client の再生成忘れ

- スキーマに `comment` カラムを追加してマイグレーションを当てた
- `prisma generate` を忘れてクライアントの型が古いまま
- → 実行時に `Unknown argument` エラー

**学び**：マイグレーションとクライアント生成はセットで意識する

---

## ハマったところ（2）

### トランザクションの書き方ミス

- 候補日の編集を `$transaction` でラップしたつもり
- コールバック内で `tx` ではなく素の `prisma` を呼んでいた
- → 実質トランザクションが効いていない
- 途中で失敗すると DB が中途半端な状態になるリスク

**現状：要修正**

---

## ハマったところ（3）

### JST タイムゾーンの扱い

- 候補日はブラウザのローカル文字列で受け取る
- サーバー側で JST 前提として UTC に変換
- 変換ロジックを `lib/datetime.ts` に一元化してテストでカバー

### 開発環境のクロスプラットフォーム問題

- npm のグローバル設定が `/bin/bash` を指していて Windows で postinstall が失敗
- 環境構築の再現性も地味だが重要

---

## 今後の改善ポイント

- トランザクションバグの修正（最優先）
- ホスト管理画面のレスポンスから `responseToken` を除外
- レート制限（予定作成APIのスパム対策）
- 候補日削除時の Answer カスケード仕様の明文化

- 実装した分だけ**テストで裏付け**を取る
