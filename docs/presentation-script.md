# 発表台本：予定調整 Web アプリ（imin-260624）

> 想定時間：5〜8分（質疑応答別）
> 対象：社内エンジニア

---

## 0. オープニング（30秒）

> 「ログイン不要で予定調整ができるWebアプリを、設計から実装・テストまで一通り作りました。今日はどう作ったか、何を一番気にしたかを話します。」

- プロダクト名：予定調整 Web アプリ（Doodle / 調整さん に近いUX）
- 一番の特徴：**アカウント登録・ログイン不要**。URLトークンだけで完結する

---

## 1. 何を作ったか（1分）

ホストが候補日を出し、ゲストがログインなしで出欠を回答できるサービスです。

- ホスト：タイトル・説明・候補日（日付 or 日時）・回答期限を設定して予定を作成
- ゲスト：共有されたURLを開いて、表示名を入れるだけで回答（出席可 / 出席不可 / 未定 + コメント）
- ホスト：管理用URLから回答状況を一覧・集計で確認、後から候補日や期限を編集可能

> 「肝は、サーバーサイドの認可をログインではなく "推測困難なトークン" でやっていることです。」

---

## 2. アクセス制御の設計（1分）— ここが一番気を付けた点

ログインを無くす代わりに、3種類のトークンで権限を分離しました。

| トークン | 役割 | 長さ |
|---|---|---|
| `eventId` | ゲストが予定を見る・回答するためのID | nanoid 21文字 |
| `hostToken` | ホストが管理画面に入るための鍵 | nanoid 32文字（ゲスト用より長い） |
| `responseToken` | ゲスト自身が後から回答を編集するための鍵（Cookieに保存） | nanoid 32文字 |

気を付けたポイント：

- **存在しないeventId・間違ったhostTokenは、すべて同じ404を返す**。「IDは存在するがトークンが違う」を漏らさない設計（情報漏えい防止）
- ホスト管理画面のAPIレスポンスから、**ゲストの`responseToken`は返さない**（ホストが他人の回答を書き換えられてしまうため、レビューで気付いて直しました）
- 回答期限が過ぎたら、サーバー側で403を返して新規回答・更新をブロック（クライアント側のチェックだけに頼らない）

---

## 3. 技術スタックと選定理由（1分）

| レイヤ | 選定 | 理由 |
|---|---|---|
| Next.js 16 (App Router) | フルスタック一体、API RoutesとUIを同じリポジトリで完結 |
| TypeScript + Zod | API入力とフォーム入力のスキーマを共通化し、型と実行時バリデーションのズレを防止 |
| Prisma + SQLite (better-sqlite3 adapter) | MVPとしてセットアップが軽量。スキーマ変更はマイグレーションで管理 |
| date-fns / date-fns-tz | 日付・日時をJST固定で扱うための変換ロジックを一箇所に集約 |
| Vitest | ユニットテスト + APIの統合テストを同じツールで実行 |

---

## 4. 実装の進め方（1分）

設計ドキュメント（`docs/design.md`）を先に作り、画面・API・データモデルを決めてから実装に入りました。

進めた順序：
1. 基盤セットアップ（Next.js + Prisma + SQLite）
2. 予定作成API・ゲスト回答・作成フロー画面
3. ホスト管理画面・予定編集API
4. 再回答モード（Cookieに保存したトークンで前回の回答をデフォルト表示）
5. Vitestでユニット・API統合テストを追加（69件）
6. 候補日入力UIをカレンダー選択＋期間一括追加に改善
7. 集計サマリー表示
8. コメント欄・「未定」ステータスの追加

> 「最初に全部の機能を作るのではなく、コア（作成→回答→管理）が動く状態を早く作って、そこからUIや細かい機能を後から足していきました。」

---

## 5. テストで気を付けたこと（1分）

- ロジック（日付変換・トークン生成・スロットの重複チェックなど）は `lib/` に切り出して、UIやAPIに依存しない形でユニットテスト
- APIは実際にPrisma経由でSQLiteに対して統合テストを実行（モックでお茶を濁さない）
- テスト用DBは本番用と分離（`test.db`、`vitest` の `globalSetup` で初期化）

---

## 6. ハマったところ・気を付けたところ（1〜2分）— 一番話したいパート

- **Prisma Clientの再生成忘れ**：スキーマに`comment`カラムを追加してマイグレーションを当てても、`prisma generate`を忘れるとクライアントの型が古いままで実行時エラーになる。マイグレーションとクライアント生成はセットで意識する必要があった。
- **トランザクションの書き方ミス**：候補日の編集（追加・更新・削除）を`$transaction`でラップしたつもりが、コールバック内で`tx`ではなく素の`prisma`を呼んでいて実質トランザクションが効いていなかった。レビューで発見、要修正の状態（途中で失敗すると中途半端な状態でDBが更新されるリスクがある）。
- **JSTタイムゾーンの扱い**：候補日の入力はブラウザのローカル文字列（"YYYY-MM-DD" や "YYYY-MM-DDTHH:mm"）で受け取り、サーバー側でJST前提でUTCに変換。タイムゾーンの変換ロジックを`lib/datetime.ts`に一元化してテストでカバーした。
- **開発環境のクロスプラットフォーム問題**：npmのグローバル設定が`/bin/bash`を指していて、Windows環境でpostinstallスクリプト（`prisma generate`）が失敗した。環境構築の再現性は地味だが重要。

## 想定質疑（準備しておくと安心）

- **Q. なぜSQLite？本番でも使う？**
  A. MVPとしての軽量さ重視。トラフィックが増えたらPostgresに移行する想定で設計ドキュメントにも明記している。
- **Q. ホストとゲストの権限分離はどう保証している？**
  A. APIレベルで`hostToken`の一致を必ずチェックし、ミスマッチは404扱い。フロントの表示制御だけに頼っていない。
- **Q. テストはどこまでカバーしている？**
  A. 日時変換・トークン生成・スロットの重複検出などのロジック関数、および主要API（作成・取得・回答・更新）の統合テスト。UIコンポーネントの単体テストは現状なし。
